# 投资会议纪要 - 多人复盘会

先读取并遵守同一 skill 包内的基础 skill：`投资会议纪要整理`；如果导入环境使用英文别名，则读取 `investment-meeting-minutes`。不要依赖 macOS 绝对路径。

跨系统导入要求：本文件必须保持 UTF-8 无 BOM、LF 换行，并通过基础 skill 的 `validate_utf8_text.py --portable-skill` 校验；Windows 导入时不得保存为 ANSI/GBK。

本类型的额外规则：

- 元信息必须保留 `**会议标题**` 和 `**会议类型**：多人复盘会`。
- 输出格式使用当前基础格式：`# 投资会议纪要` + 元信息 + `## 一、逐发言人原文整理`；只有存在真实存疑内容时，才追加 `## 二、存疑与待确认`。
- 如果本次整理提供了 `analysis_ledger`，只把它作为内部依据；类型 skill 仍是最终格式权威，主 Agent 仍是唯一写作者，Reviewer 结果只用于发现问题；不得把 ledger 字段、JSON、角色名或处理过程写入终稿正文。
- 不同投资动作优先拆段；标题必须覆盖明确推荐对象；背景或比较对象不得占据推荐标题。
- 首要原则是最大限度保留每位发言人的原文和真实表达，只剔除无意义语气词、无意义重复和明显 ASR 噪声；不得把复盘内容压缩成简短摘要。
- 发言人原文部分必须保留原发言人视角和人称，尤其保留“我/我们/这边/咱们”等第一人称表达；不得把“我看好/我减仓/我们明天对接”等改写成“某老师看好/其减仓/团队将对接”这类第三人称旁白。
- 不要用整理者自己的语言重写发言人原文。除断句、去口水词、纠明显 ASR 错词和拆分主题外，尽量保留发言人的原始措辞、语序、判断边界和口语化表达；不得把原文改写成更“像研报”的表述。
- `## 一、逐发言人原文整理` 严格按实际发言顺序排列。
- 同一发言人多次出现时，不合并到第一次发言下；保留在原位置，可写为 `发言人X（后半段）`。
- 每个发言人下面按 `【板块｜标的(代码)】` 拆段；一个小段只承载一个标的或一个纯板块主题。
- 每个小段标题必须覆盖本段真实主标的；如果存在明确推荐、看多、看空、重点关注、买入、卖出、加仓或减仓对象，标题或正文开头必须体现真实推荐标的。
- 一段复盘同时涉及多个标的时，先区分主标的、推荐标的、比较对象、客户/供应商/竞争对手、上下游相关方、行业背景和顺带提及对象；不得因为某个标的先出现或出现次数多就把它当作推荐标的。
- 如果主标的和推荐标的不一致，应拆成多个小段，或在标题中同时覆盖二者；无法判断主次时，标题应体现多标的并将归因问题写入存疑。
- 如果复盘会中出现问答内容，该问答局部使用专家交流模板：问题加粗，回答以 `【】` 主题标签开头，标签后直接接回答正文，不写 `A：` 或 `A:`；标签不限定字数但必须概括回答主旨方向。
- 保留市场节奏、仓位动作、高低切、买卖计划、催化、利润/市值/估值数字、犹豫和条件表达。
- 对所有存疑内容同时结合音频/转录、上下文、`a-stock-data` 标的证据和专业名词搜索核验；行业术语、产品名、技术路线、政策/事件词、机构/人物名称不能只靠 ASR 猜测。若搜索证据、会议上下文和音频听感无法统一，保留原文表述并写入存疑表。
- 整理后的每条真实存疑都必须完成上下文 + 搜索/证据核验，并在 `核验依据` 列写明 `上下文：...；检索/证据：...`，不能只写“待确认”或只依赖 ASR。
- 对标的归因、推荐对象、公司名、代码、术语、客户名、订单、产能、价格、收入、利润等高风险信息，不能确认时必须进入存疑表，并给出上下文和证据路径。
- 所有存疑片段必须在原文中同步标注：Markdown 中用 `**存疑词**` 加粗。只有当前会话含音频/视频或带时间戳转写时，才在加粗片段后立即写明 `（存疑时间戳：HH:MM:SS）`；文稿/文档-only 会话正文不写时间戳占位，但存疑表 `时间戳` 单元格写 `未提供`。导出 Word 时存疑词必须显示为加粗 + 下划线；这是所有会议类型都要遵守的基础规则。
- `## 二、存疑与待确认` 只在存在真实存疑内容时输出；表头固定为 `时间戳 | 原始表述 | 当前判断 | 存疑原因 | 候选项 | 核验依据 | 人工确认`。文本/文档来源或无法定位时间锚点时，`时间戳` 写 `未提供`；`人工确认` 单元格留空供用户填写。
- 不输出结构化摘要和标的表格；这些只在人工确认终稿后由后续脚本生成。
