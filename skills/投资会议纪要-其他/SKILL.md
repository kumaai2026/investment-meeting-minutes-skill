# 投资会议纪要 - 其他

先读取并遵守同一 skill 包内的基础 skill：`投资会议纪要整理`；如果导入环境使用英文别名，则读取 `investment-meeting-minutes`。不要依赖 macOS 绝对路径。

跨系统导入要求：本文件必须保持 UTF-8 无 BOM、LF 换行，并通过基础 skill 的 `validate_utf8_text.py --portable-skill` 校验；Windows 导入时不得保存为 ANSI/GBK。

本类型的额外规则：

- 元信息必须保留 `**会议标题**` 和 `**会议类型**`，如果用户填写了自定义类型，使用自定义类型原文。
- 首要原则是最大限度保留原文，只剔除无意义语气词、无意义重复和明显 ASR 噪声；不得为了套格式压缩原文信息密度。
- 如果原文包含问答内容，该部分必须使用 `专家交流` 的问答模板：问题加粗，回答以 `【】` 主题标签开头，标签后直接接回答正文，不写 `A：` 或 `A:`；标签不限定字数但必须概括回答主旨方向。
- 如果原文不是访谈，而是汇报、讨论或复盘，按原始形态和实际顺序整理；不要强行把所有内容改写成问答。
- 允许保留访谈、复盘、汇报、讨论、纪要等不同形态的信息密度，不得删除有意义的观点、数字和不确定表达。
- 对所有存疑内容同时结合音频/转录、上下文、`a-stock-data` 标的证据和专业名词搜索核验；行业术语、产品名、技术路线、政策/事件词、机构/人物名称不能只靠 ASR 猜测。若搜索证据、会议语境和音频听感无法统一，保留原文表述并写入存疑表。
- 所有存疑片段必须在原文中同步标注：Markdown 中用 `**存疑词**` 加粗，导出 Word 时必须显示为加粗 + 下划线；这是所有会议类型都要遵守的基础规则。
- 不输出结构化摘要和标的表格；这些只在人工确认终稿后由后续脚本生成。

输出骨架：

```markdown
# 投资会议纪要

**会议日期**：YYYY-MM-DD
**整理时间**：YYYY-MM-DD
**会议标题**：原始会议标题
**会议类型**：用户选择或自定义类型

---
```
