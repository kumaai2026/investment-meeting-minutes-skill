# 输出合约

本文件是最终 Markdown 和 Word 纪要的格式唯一来源。生成、校验和导出都以本合约为准。

## 目录

## Markdown 结构

```markdown
# 投资会议纪要

**会议日期**：YYYY-MM-DD
**整理时间**：YYYY-MM-DD
**会议标题**：原始会议标题
**会议类型**：多人复盘会 / 上市公司交流 / 专家交流
**会议系列**：研究所周会
**会议标的**：一级板块｜细分板块｜标的(代码)  # 仅上市公司交流必填

---

## 一、发言整理

### 张三

#### 【中芯国际(688981.SH)｜北方华创(002371.SZ)】
##### 【科技｜半导体】
中芯国际这一段景气度还是在往上走，订单能见度比之前好。设备端的北方华创相对更早感受到资本开支恢复。

---

## 二、存疑与待确认

音频/视频或带时间戳转写：

| 时间戳 | 原始表述 | 当前判断 | 候选项 | 人工确认 |
| --- | --- | --- | --- | --- |
| 12:34 | 某某科技 | 待人工确认 | 候选A(000001.SZ) / 候选B(09999.HK) | |

文稿/document-only：

| 原始表述 | 当前判断 | 候选项 | 人工确认 |
| --- | --- | --- | --- |
| 某某科技 | 待人工确认 | 候选A(000001.SZ) / 候选B(09999.HK) | |
```

只有存在真实存疑内容时才输出 `## 二、存疑与待确认`；无存疑时整节省略，不写“暂无存疑”。

## 硬性规则

- `## 一、发言整理` 只放按发言人和真实发言顺序整理的校对后原文，不放分析项目符号、汇总表格或整理者摘要。
- 发言内容必须按实际发言顺序排列；同一发言人多次出现时保留在原位置，不使用 `（后半段）` 合并标识。
- 三级发言人标题优先使用当前会议内容可确认的人名或真实角色；可用证据包括自我介绍、主持人称呼、议程角色、问答角色或稳定转写标签。无法可靠识别时，按首次出现顺序使用 `发言人1`、`发言人2`、`发言人3` 等，不为显得完整而猜测身份。
- 所有会议类型都必须保留发言人的视角、人称、判断边界、口语化表达和原始逻辑；不要改写成研报口吻、结论列表、摘要或压缩版纪要。
- 原文使用 `我`、`我们`、`个人觉得`、`我的判断`、`我们这边` 等第一人称或说话方视角时，终稿正文必须保留同一视角；不得为了显得正式而改成第三人称转述。
- 不把原始回答默认改成“专家表示/认为”“管理层强调/表示”“公司表示”等第三人称转述口吻；除非原始材料本身就是这样表述。
- 只允许删除纯口水词、无意义重复、明显 ASR 噪声和重复起手式；不得删掉判断、理由、数字、时间点、催化、业绩预期、仓位动作、条件表达、主观不确定性或说话人的语气边界。
- 不得为了显得正式、简洁或有条理而重写原句、合并多轮发言、改变因果顺序、补充整理者判断或把口语改成研报式表达。
- 长回答可以分段，但每段都应来自同一轮发言的连续或相邻原文；不得把多轮发言压缩成 `主要包括`、`核心观点`、`总结来看`、`整体判断` 之类整理者概括。
- 不输出 `摘要`、`投研摘要`、`整理者总结`、`标的汇总表`、`投资结论` 等替代原文整理的章节。
- 不输出契约外逃逸标题，例如 `发言片段`、`未归类`、`主题整理`、`内容摘要`、`观点汇总`、`核心观点`、`整体判断`。成稿前必须按本文件和会议类型 reference 做标题层级自检。
- 不输出流程字段或整理阶段说明，例如 `输入来源`、`整理说明`、工具名、日志、路径、临时工作流链接或临时标识。
- 问答内容中，问题用 `**【问题原文】**`；不使用 `Q：` / `提问：` / 裸 `A:` / `A：`。

## 标题与标的

- 多人复盘会的小段标题以对应 reference 为准，先写标的行，再写低一级板块标题。
- 发言人标题不是标的或主题标题；不要把 `科技`、`存储`、`AI硬件`、`核心观点` 等主题词写成三级发言人标题。
- 无明确证券标的时，保留空标的行，不使用主题或板块强行填充。
- 标题中的板块需要结合上下文自行归因到一级板块，再写细分板块；不要只机械摘录原文。
- 小段标题块必须覆盖该段提到的全部证券标的；多个标的共享同一逻辑链时放在同一个标的标题行。
- 只有当不同标的对应独立投资动作、独立逻辑链或容易混淆归因时，才拆成多个小段。
- A 股、港股、美股、ADR、台股等证券标的都应尽量补足代码和交易所后缀。核验口径见 `verification_policy.md`。

## 存疑

- 原文中存在不确定识别的词、公司名或术语时，正文必须对存疑片段加粗；导出 Word 时存疑词必须同时加粗并加下划线。
- 含音频/视频或带时间戳转写且时间戳可靠时，正文存疑片段后立即标注 `（存疑时间戳：MM:SS）`、`（存疑时间戳：HH:MM:SS）` 或同格式范围；可靠时间戳只接受 sentence/phrase 级 anchor，或 `source=sensevoice_vad_segment` 且 `duration_ms <= 10000` 的短 VAD segment。文稿/document-only、无可靠时间戳，或只有其他 segment/chunk/minute 级范围时，只加粗存疑词，不写时间戳占位。
- 存疑表列固定：有可靠时间戳时用 `时间戳 | 原始表述 | 当前判断 | 候选项 | 人工确认`；无可靠时间戳或 document-only 时用 `原始表述 | 当前判断 | 候选项 | 人工确认`。
- 不写 `未提供` 时间戳列；没有可靠时间戳就使用无时间戳表格。
- `人工确认` 列必须留空。
- 生成终稿前先建立内部 `doubtful_items` 清单，字段见 `verification_policy.md`。存疑词反查先用同一清单匹配 `timestamp_index.text`，再派生正文 inline 时间戳和存疑表时间戳列。终稿 `## 二、存疑与待确认` 只能从该清单派生固定列，不加入 `上下文依据`、`检索/证据路径`、`最终处理` 等核验过程字段。
- 非人名业务存疑项仍需在生成前按 `verification_policy.md` 做上下文核验。`身份`、`嘉宾`、`发言人` 等词只有在该行确实是纯人名或纯说话人身份不确定时才按人名类处理；公司或证券标的、行业术语、数字或时间、其他业务事实不得因此跳过核验。
- 需要审计时，在终稿同目录生成同 stem 的 `.verification.json` 或 `.verification.jsonl`。sidecar 也只能从同一批 `doubtful_items` 中 `是否需要 sidecar=true` 的记录派生，记录终稿非人名存疑行背后的核验过程，不得二次改写 `原始表述`、`当前判断`、`候选项` 或 `最终处理`。
- 交付前做一次人工式自检：sidecar 中每条记录都能回到终稿的一条非人名存疑行；同一条的 `原始表述`、`当前判断`、`候选项`、`最终处理` 不互相冲突。若冲突，回到 `doubtful_items` 统一修正并重新派生终稿表格和 sidecar，不把这类语义一致性新增为 validator hard rule。
- 候选项如能匹配明确证券代码，可写为 `标准名称(代码.市场)`，并同步放入多人复盘会标的标题或正文存疑片段；只要原始表述仍未完全确认，正文仍保留加粗存疑标记，存疑表也不得省略。

## 会议类型差异

三类会议共享本文件的元信息、存疑、Word 导出和禁止项规则。正文格式必须同时参考对应分类型 reference：

- `多人复盘会`：`meeting_types/review_meeting.md`
- `上市公司交流`：`meeting_types/listed_company.md`
- `专家交流`：`meeting_types/expert_call.md`

### 多人复盘会

- 默认会议类型；无法明确判断为上市公司交流或专家交流时使用本类型。
- 适用于盘前交流、多人复盘、晨会/周会/例会，以及多位发言人依次分享投资信息、投资逻辑、持仓、仓位动作和市场节奏。
- 严格按实际发言顺序排列；同一发言人多次出现时保留在原位置，不按主题或标的重新汇总。
- 同一连续发言内按不同标的或板块分段时，不重复输出三级发言人标题。
- 小段标题层级以 `meeting_types/review_meeting.md` 为准。

### 上市公司交流

- 仅用于某家公司的专场会议，讨论内容集中于该公司经营、业务、业绩、订单、产能、管理层问答和风险。
- 文章标题使用 `XX公司交流会议`。
- 必须输出 `**会议标的**：一级板块｜细分板块｜标的(代码)`。
- 正文结构以 `meeting_types/listed_company.md` 为准，不得改写为公司研究报告。

### 专家交流

- 仅用于专家问答形式的交流；讨论对象可以是某家公司，也可以是某个产业或产业链主题。
- `## 一、发言整理` 必须采用问答形式，正文结构以 `meeting_types/expert_call.md` 为准。
- 保留专家判断依据、数据来源、产业链位置、客户反馈、渠道口径、产能/订单/价格/利润等，不把长回答压缩成短摘要。

## Word 导出

- Word 文档顺序：标题区 `投资会议纪要`、元信息表、`一、发言整理`、真实存疑存在时追加 `二、存疑与待确认`。
- 不把 Markdown/YAML 符号、上传文件名、工具日志、路径等流程信息写入 Word。
- 元信息表和存疑表必须使用真正 Word 表格；表头使用深蓝底色、白字，正文行可使用浅色隔行底纹。
- 多人复盘会的标的标题和板块标题分行展示，避免挤成一行。
- Markdown 中的 `**存疑词**` 导出到 Word 后必须加粗并加下划线。
- 最终交付不生成 PDF。PDF 输入只作为附件归档，正文整理应使用音频、DOCX、TXT、Markdown 或用户另行提供的可读文本。

## 归档

原始输入、最终文件命名、输出目录和历史文件处理以 `archive_naming_contract.md` 为准。
