# 核验与标的归因规则

本文件合并名称校对、股票代码来源、证据边界和标的归因规则。外部资料只能用于确认名称、代码、术语、公开客户/供应商、事件或数字口径，不得补写会议没有出现的新结论。

## 证据层级

### source_evidence

来源包括原始音频、SenseVoice 转写、带时间戳转写 JSON、用户上传文档、人工初审稿和同会话补充说明。

地位：
- 是会议内容最高优先级证据。
- 只能据此整理会议中实际说过的判断、数字、动作和不确定表达。
- 外部资料不能替代 source evidence 编造会议结论。

### local_reference_evidence

来源包括 `query_symbol_candidates.py`、本地 A/HK/US 代码表、公司别名表、本地术语表或历史归档中可用的候选信息。

地位：
- 只用于候选纠错和缩小检索范围。
- 本地候选不是最终确认。
- 只有本地候选、外部尚未确认时，不得标记 `confirmed`。
- 置信不足、候选冲突或资源过期时必须保留为存疑或继续外部核验。

### external_verification_path

来源包括公司公告、交易所披露、CNINFO、公司官网、监管/协会资料、行业报告、可靠财经新闻和 `a-stock-data` live/network data。

地位：
- 辅助确认实体、代码、术语、客户公开信息、订单/产能/价格口径。
- 与 source evidence 冲突时，保留原始表述并列为存疑。

## 名称和代码核验

本地资源目录：

`$INVESTMENT_MINUTES_WORKSPACE/03 Resources`

常用路径：
- `market-symbols/a_share_list.csv`
- `market-symbols/global-stock-symbols-source/`
- `market-symbols/README.md`

查找顺序：
1. `a-stock-data` live sources when available.
2. Local A-share table.
3. Local global-symbol table for HK/US/other markets.
4. Official exchange, company announcement, company website, or other professional source when local candidates are insufficient.

规则：
- 不要把未确认 ticker 写成确定标题代码。
- 本地 symbol candidates 在多候选匹配时不是最终证据。
- 不局限于 A 股；HK/US/ADR/TW/其他上市证券作为会议标的、客户、供应商、竞争对手或重要可比对象出现时也要核验。
- 市场后缀保持一致：A 股优先 `.SZ`、`.SH`、`.BJ`；港股 `.HK`；美股使用本地/global symbol table 中的交易所 ticker。
- 当存疑口播名称有可编码候选时，最终纪要可把候选写成加粗存疑形式，例如 `**候选公司(09999.HK)**`，同时保留存疑表人工确认入口。

## 高风险 claim

以下信息写入终稿前必须核验：
- 公司名和股票代码
- 客户名、供应商名、竞争对手名
- 订单、产能、价格、收入、利润、毛利率、估值、市值
- 时间节点、政策事件、公告事件
- 产品型号、技术路线、专业术语
- 投资动作：买入、卖出、加仓、减仓、重点跟踪

`candidate_only` / `unverified` / `not_found` 的 ticker 不得作为已确认代码写入最终标题。若候选项能够匹配到明确证券代码，可在正文或标题中以加粗存疑形式保留候选名称和代码，同时在存疑表中保留人工确认入口。

## 存疑处理

先建立一份内部 `doubtful_items` 清单，再从同一清单派生终稿表格和 verification sidecar。不要让 sidecar 变成第二套独立判断。

每条 `doubtful_items` 必须包含：
- `原始表述`：ASR、文稿或用户材料中的原始存疑词句。
- `存疑类型`：只能使用 `人名`、`说话人身份`、`公司或证券标的`、`行业术语`、`数字或时间`、`其他业务事实`。
- `当前判断`：写入终稿表格。
- `候选项`：候选为证券标的且能匹配代码时，写标准名称和代码。
- `是否需要 sidecar`：纯人名或纯说话人身份不确定为 `false`；公司或证券标的、行业术语、数字或时间、其他业务事实为 `true`。
- `上下文依据`：记录相关原文段落、语义位置、发言人/版块或时间戳线索。
- `检索/证据路径`：记录本地候选、a-stock-data、公告、交易所、官网、行业资料或其他可靠资料路径。
- `最终处理`：记录写入正文、正文加粗标注并入存疑表、不写入正式标的等处理。

`身份`、`嘉宾`、`发言人` 等词只在该行确实是纯人名或纯说话人身份不确定时，才按人名类处理。若疑点实际指向公司、证券标的、行业术语、数字、时间或其他业务事实，仍按对应业务类型进入核验流程。

不得只写 `待确认`、`需核实`、`未查到`、`听不清`；这些词不能替代当前判断和候选项。
需要审计时，只把 `是否需要 sidecar=true` 的同一批 `doubtful_items` 派生到终稿同 stem 的 `.verification.json` 或 `.verification.jsonl`。核验依据不写入最终 `## 二、存疑与待确认` 表格。

交付前做一次人工式自检：
- sidecar 中每条记录都能回到终稿的一条非人名存疑行。
- 同一条的 `原始表述`、`当前判断`、`候选项`、`最终处理` 在终稿表格和 sidecar 中不互相冲突。
- 如果冲突，回到 `doubtful_items` 统一修正并重新派生终稿表格和 sidecar，不通过新增 validator 硬规则解决。

当出现非人名存疑项时，主流程或 subagent 必须使用同一套核验 prompt：

```text
你是中文投研会议纪要的存疑项核验助手。请只核验当前会议材料中已经出现的非人名存疑项，不补写会议没有说过的新结论。

输入：
1. 会议上下文：相关原文段落、发言人/版块、时间戳或来源位置。
2. 存疑原始表述：ASR 或文稿中的原始词句。
3. 存疑类型：人名 / 说话人身份 / 公司或证券标的 / 行业术语 / 数字或时间 / 其他业务事实。
4. 候选解释：可能的人名、说话人身份、公司名、股票代码、简称、英文缩写、客户/供应商、数字、日期、产品、政策或事件。

核验要求：
1. 先用会议上下文判断候选是否语义匹配。
2. 再使用至少一种可靠证据路径核验：联网搜索到的可靠来源、a-stock-data、公司公告、官网、交易所/监管披露、研报/行业资料。本地代码表只作为候选线索，不能单独作为确认依据。
3. 本地候选和 ASR 输出只能作为线索，不能单独作为确认依据。
4. 外部资料只能用于确认名称、代码、术语、公开客户/供应商、事件或数字口径，不能补写会议没有出现的观点。
5. 如果证据冲突、证据不足或无法唯一确认，保留原始表述并进入存疑表。
6. `身份`、`嘉宾`、`发言人` 等词只在疑点确实是纯人名或纯说话人身份时按人名类处理；非人名业务疑点不得因此跳过核验。

输出每条 `doubtful_items`：
- 原始表述：
- 存疑类型：人名 / 说话人身份 / 公司或证券标的 / 行业术语 / 数字或时间 / 其他业务事实
- 当前判断：确认 / 保留原文 / 暂不采信 / 待人工确认
- 候选项：
- 是否需要 sidecar：true / false
- 上下文依据：
- 检索/证据路径：
- 最终处理：写入正文 / 正文加粗标注并入存疑表 / 不写入正式标的

注意：终稿 `## 二、存疑与待确认` 和 verification sidecar 都只从同一份 `doubtful_items` 派生；sidecar 记录核验过程，不二次改写判断。最终 Markdown 表格保留固定列，不加入详细核验依据。
```

## 校对规则

- 修正明显 ASR 错词，但保留发言人的原始判断、条件、语气和不确定表达。
- 公司全称、简称、ticker、英文缩写和行业术语要结合上下文、代码资源和外部证据核验。
- 不因外部资料更完整就覆盖音频/文稿中的明确不确定表达。
- 纯人名或普通发言人标签不确定时可忽略，除非身份改变投资事实、标的归因或 source credibility。
- 不用公开资料补写原文没有的客户名、订单额、业绩预测。
- 不把传闻、小作文、未公开客户名写成确定事实。

## 标的归因

推荐标的、比较标的、客户、供应商、竞争对手、上下游、产业背景和偶然提及对象要区分处理。

核心规则：
- 最先提到或频次最高的标的不自动等于主标的或推荐标的。
- 客户、供应商、竞争对手、可比公司、上下游实体和背景对象，没有明确 source evidence 时不得提升为推荐标的。
- 负向动作不是推荐，但动作对象仍要记录准确。
- 如果讨论对象和动作对象不同，标题应覆盖两者；只有逻辑链或动作独立时才拆段。
- `primary_target` 和 `recommendation_target` 通常应进入标题；`ticker_status` 非 confirmed 时不得作为无标记确认代码写入标题。

强推荐信号包括：推荐、重点推荐、核心推荐、首选、第一选择、可以买、建议买入、准备买、准备加仓。

弱正面或跟踪语言不得自动变成推荐标的，例如：建议关注、重点跟踪、弹性最大、空间较大、赔率不错、位置不错、值得重视、可以看看、有机会、短期催化。

动作映射：
- “没有减仓”“继续拿着”：`hold`
- “减仓”：`reduce`
- “不建议追”“先回避”：`avoid`
- “看空”：`negative`
- 纯事实背景：`neutral`
- 来源不清：`unclear`
